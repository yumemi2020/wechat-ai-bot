測試版 v0.12

WeChat AI Bot 操作手冊

本系統透過 WeChatPadProMAX 接收微信訊息，由本地大語言模型（LM Studio 等）自動回覆， 並提供網頁控制台管理聯絡人、人設、圖片庫與長期記憶。文件更新日期：2026-06-26。

1. 系統概覽

整體資料流如下：

微信好友 / 群組發訊息 ↓ WeChatPadProMAX（埠 1238）→ Webhook POST ↓ 後端 FastAPI（埠 9950）→ 寫入 MariaDB + WebSocket 推播前端 ↓ （若 AI 開啟且為文字訊息）本地 LLM（埠 1234）生成回覆 ↓ WeChatPadProMAX API 發送回微信
元件技術預設埠說明
後端FastAPI + asyncmy + MariaDB9950main.py，Webhook 與全部 API
前端Vue 3 + Vite + Tailwind9960網頁控制台（Docker 對外埠，內部 5173）
資料庫MariaDB wechat_ai3306（內部）對話、人設、聯絡人設定持久化
本地 LLMOpenAI 相容 API1234LM Studio / Ollama 等
微信閘道WeChatPadProMAX1238收發訊息、Webhook 推送

2. 架構與服務

Docker 三服務（建議部署方式）

  • wechat_ai_db — MariaDB，資料目錄 db_data/
  • wechat_ai_bot — Python 後端，使用根目錄 Dockerfile 建置（含 gcc、ffmpeg、pilk），掛載專案至 /app
  • wechat_ai_frontend — Vite 開發伺服器，掛載 frontend/
後端首次部署或更新 requirements.txt 後，請執行 docker compose up -d --build wechat_ai_bot 重新建置映像。

即時推播

前端透過 WebSocket /ws/frontend 接收新訊息與媒體更新（new_message、message_updated），無需手動重新整理對話列表。

背景任務

  • AI 回覆：Webhook 收到文字訊息後即時寫庫，LLM 思考在背景執行。
  • 圖片 / 語音下載：Webhook 先寫佔位文字並立即回 200，避免 PadPro 逾時重試；背景下載完成後更新 DB 並推播前端。
  • 記憶萃取：每累積 20 則訊息，背景呼叫 LLM 更新長期記憶摘要。

3. 日常操作流程

微信掃碼登入 / 登出（v0.09+）

  1. 點控制台 🛠️ 系統設定。
  2. 登入：點「產生 QR Code 掃碼登入」，掃碼後自動 Newinit + 註冊 Webhook。
  3. 切換帳號：先點「登出微信帳號」→ 再產生新 QR Code 掃碼。
  4. 登出時系統會呼叫 PadPro /Login/LogOut，並將 Webhook 設為 enabled: false。

啟用某位好友的 AI 代答

  1. 確認頂部全域 AI 開關為開啟（綠色）。
  2. 左側點選聯絡人，右側開啟「允許 AI 代答」開關。
  3. 新聯絡人預設為「一般名單」（AI 關閉），需手動開啟。
  4. 群組（@chatroom）新建時同樣預設關閉 AI。

手動介入回覆

當全域 AI 關閉，或該聯絡人 AI 關閉時，底部會出現手動輸入框。可發送文字、從圖片庫選圖，或 Ctrl+V 貼上截圖。

AI 大模型設定（v0.11+）

  1. 點控制台 🛠️ 系統設定，開啟「🤖 AI 大模型設定」面板。
  2. 可新增多組 LLM 設定（OpenAI / DeepSeek / Groq / Ollama / 自訂），並標示「使用中」的啟用項。
  3. 切換啟用後，下一次 AI 回覆與記憶萃取即套用新模型；無需重啟後端。
  4. 首次啟動若資料庫尚無設定，會自動從 .env 的 LOCAL_API_* 匯入一筆並設為啟用。

測試 LLM 連線

點頂部「測試 AI 連線」，確認目前啟用中的模型可正常回覆。若失敗，請在系統設定檢查 Base URL、Model Name 與 API Key。

重啟 / 重建後端

# 僅重啟（改 main.py，有 --reload 時通常足夠）
docker restart wechat_ai_bot

# 更新 requirements.txt 或 Dockerfile 後重建
docker compose up -d --build wechat_ai_bot

5. 圖片與語音訊息（v0.09）

圖片（msgType = 3）

  1. Webhook 收到後先寫入 [圖片訊息] 佔位，立即回應 PadPro。
  2. 背景依序嘗試 CdnDownloadImage、DownloadImg 下載。
  3. 存檔至 /app/photos/received/，DB 寫入 [IMAGE_MSG]/api/media/received/{token}/file。
  4. 前端透過 message_updated 自動替換為可顯示圖片；點擊可全螢幕預覽。

語音（msgType = 34）

  1. Webhook 收到後先寫入 [語音訊息] 佔位。
  2. 背景依序嘗試 Webhook base64、CdnDownloadImage（voiceurl）、DownloadVoice。
  3. 以 pilk 解碼 SILK，再經 ffmpeg 轉為 MP3。
  4. 存檔至 /app/photos/received/voices/，DB 寫入 [VOICE_MSG]/api/media/received/{token}/file。
  5. 前端顯示 <audio controls> 播放條。
注意：圖片與語音訊息不觸發 AI 自動回覆（與文字訊息分流）。語音轉文字（ASR）尚未實作。

自己從手機發出的圖片 / 語音

標記為 [手機回覆]，同樣在背景下載並更新顯示，不觸發 AI。

6. AI 回覆機制

雙重驗證

必須同時滿足：全域 AI 開 + 該聯絡人 ai_enabled = true，且訊息為文字（msgType=1），才會觸發自動回覆。

System Prompt 組裝順序

  1. 當前時間感知（依 APP_TIMEZONE，預設台北）
  2. 長期記憶摘要（若有 memory_summary）
  3. 聯絡人綁定人格，或全域啟用人設
  4. 該人格專屬照片庫目錄（供 AI 使用 [發送照片: 關鍵字]）

短期對話上下文（滑動視窗）

每次呼叫 LLM 前，從 messages 表讀取最近 30 則訊息（SLIDING_WINDOW_MESSAGE_LIMIT），轉為多輪對話格式。圖片 / 語音標記會轉為「已發送圖片 / 已發送語音」文字描述。

三大防護網

  • 滑動視窗：避免 context 過長。
  • 斷線救援：LLM 失敗時發送備用台詞，不讓對話靜默。
  • 圖片空包彈：AI 標記要發的圖若檔案不存在，自動抹除標籤避免錯誤。

Webhook 自我訊息防護

從手機發出的訊息會寫入 DB 並標記 [手機回覆]，絕不觸發 AI，避免無限迴圈。

訊息特殊標記

標記說明
[手機回覆]您從手機親自發出的訊息
[網頁回覆]控制台手動發送
[wxid_xxx]:群組內發言者標註
[IMAGE_MSG]/api/...圖片訊息，前端渲染為圖片
[VOICE_MSG]/api/...語音訊息，前端渲染為播放器
[圖片訊息] / [語音訊息]下載中的佔位文字，完成後由 WebSocket 更新
[發送照片: 關鍵字]AI 內部標記，後端攔截後發圖並從文字移除

7. 長期記憶

自動萃取

  • 當某位聯絡人訊息總數達 20、40、60… 則時，背景任務自動執行。
  • 讀取舊記憶 + 最近 20 則對話，由獨立分析 prompt 萃取個人資訊（職業、喜好等）。
  • 結果寫入 contact_settings.memory_summary（約 100 字內）。

手動編輯

在聯絡人詳情區的「長期記憶摘要」文字框編輯後，點「儲存記憶」即可覆寫資料庫內容。

資料庫查詢範例

SELECT wx_id, nickname, memory_summary
FROM contact_settings
WHERE memory_summary IS NOT NULL;

8. 人設與照片庫

多人設庫

可建立多個 AI 人格（名稱 + System Prompt），其中一個為「全域啟用」。未單獨綁定的聯絡人使用全域啟用人設。

聯絡人專屬人格

右欄「綁定 AI 人格」下拉選單可為每位客人指定專屬人格；選「使用全域預設人設」則跟隨全域啟用者。

照片庫與人格隔離

  • 上傳圖片時需選擇「所屬 AI 人格」。
  • AI 發圖時只會從該聯絡人當前人格的照片庫中選圖。
  • 已發送過的圖片 ID 會記錄在 sent_photo_ids，避免重複發同一張。

10. 環境變數（.env）

變數說明
LOCAL_API_BASE首次啟動種子用 LLM API 位址（之後請改在 UI 管理）
LOCAL_API_KEY首次啟動種子用 API Key（可留空，如 Ollama）
LOCAL_MODEL_NAME首次啟動種子用模型名稱
PADPRO_URL / PADPRO_AUTH_KEY / PADPRO_WXID微信閘道設定
PADPRO_WEBHOOK_PUBLIC_URL掃碼登入後自動註冊的 Webhook 公網 / 區網 URL
DATABASE_URLMariaDB 連線字串（asyncmy）
SLIDING_WINDOW_MESSAGE_LIMITLLM 短期上下文則數（預設 30）
MEMORY_EXTRACTION_INTERVAL自動記憶萃取間隔（預設每 20 則）
SYNC_POLL_ENABLEDPadPro 輪詢同步（預設 false，開啟可能導致斷線）
APP_TIMEZONE時間感知時區（預設 Asia/Taipei）
PHOTOS_DIR照片與接收媒體目錄（預設 /app/photos）

前端 API 位址：frontend/.env 的 VITE_API_BASE（預設 http://127.0.0.1:9950，部署時請改為實際伺服器位址）。

11. 常見問題

API 404（例如 assigned_profile）

後端容器可能仍在跑舊程式。執行 docker restart wechat_ai_bot，或確認 uvicorn 有 --reload。

AI 不回覆

  • 檢查全域 AI 與聯絡人 AI 開關是否都開啟。
  • 確認訊息為文字；圖片、語音不會觸發 AI。
  • 點「測試 AI 連線」確認 LLM 服務正常。
  • 查看後端日誌是否有 Webhook 收到訊息的記錄。

圖片 / 語音只顯示佔位文字

  • 查看日誌是否有 無法下載客人圖片 或 無法下載客人語音。
  • 語音轉檔需容器內已安裝 ffmpeg 與 pilk；請執行 docker compose up -d --build。
  • 確認 PadPro 閘道與 PADPRO_AUTH_KEY 正常。

PadPro Webhook 逾時重試

v0.09 已改為媒體背景下載；若仍出現重試，請確認後端為最新版且 Webhook 能在 1 秒內回 200。

pilk 安裝失敗（gcc not found）

請使用專案根目錄 Dockerfile 建置後端，內含 gcc、g++、python3-dev。勿僅用裸 python:3.10-slim 映像直接 pip install。

記憶沒有自動更新

需累積滿 20 則訊息才會觸發；可手動在 UI 編輯，或查資料庫 memory_summary 欄位。

微信斷線

請保持 SYNC_POLL_ENABLED=false，避免頻繁呼叫 PadPro Sync API。
